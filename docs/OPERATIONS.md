## Local Development Setup

Bootstrap a new developer environment:

```bash
chmod +x setup.sh
./setup.sh          # installs pnpm if missing, installs deps, starts Supabase, applies migrations
```

If `setup.sh` installed `pnpm`, refresh shell first:
```bash
source ~/.zshrc
```

Manual steps (if not using `setup.sh`):
```bash
pnpm install                      # install all workspace deps
pnpm db:start                     # start local Supabase containers
pnpm db:reset                     # apply all migrations + seed
pnpm gen:types                    # regenerate TypeScript types from schema
pnpm dev                          # run all apps in parallel
```

## Database Migrations

### Naming Convention

```
supabase/migrations/YYYYMMDDHHMMSS_description.sql
```

All timestamps are UTC. Description uses underscores. Example: `20260401215434_initial_schema.sql`.

### Creating a Migration

```bash
pnpm supabase migration new <description>
# Creates: supabase/migrations/<timestamp>_<description>.sql
```

Write SQL in the generated file. Include `IF NOT EXISTS` guards where appropriate. Never modify existing migration files — create a new one for changes.

### Applying Migrations

```bash
pnpm db:reset     # wipes local DB and applies all migrations in order
```

**When to run `db:reset`:** after pulling new code that adds migrations, or when local DB state has diverged from the migration history.

### Type Regeneration

After any schema change:

```bash
pnpm gen:types
# Runs: supabase gen types typescript --local | tail -n +2 > packages/core/src/schema/database.types.ts
```

`tail -n +2` strips the CLI update notice that otherwise leaks into the TypeScript file and causes parse errors.

### Production Migrations

Migrations are applied to production Supabase via the Supabase CLI with the production project linked:

```bash
supabase db push --linked    # applies pending migrations to linked production project
```

**Pre-migration checklist:**
- Run `db:reset` locally with the migration applied — confirm no errors
- Check if migration is destructive (column drop, type change, enum removal) — run during low-traffic window
- Verify RLS policies still enforce after schema changes
- Test affected queries after migration

## CI/CD Pipeline

GitHub Actions. Defined in `.github/workflows/`.

### Triggers

- Push to `main` or `develop`
- Pull request to `main` or `develop`

### Jobs

Three parallel jobs; `e2e` depends on `test`.

**`test`** — runs on every push/PR:
```
pnpm install --frozen-lockfile
turbo run typecheck      # tsc --noEmit on @stashflow/core + @stashflow/api
pnpm lint                # tsc --noEmit on apps/web
pnpm test:coverage       # vitest with coverage; fails below threshold
```

**`security`** — runs on every push/PR:
```
pnpm audit --audit-level=high   # fails on high/critical CVEs
gitleaks detect --redact        # full history secret scan (v8.27.2)
```

**`e2e`** — runs only on PRs to `develop`, after `test` passes:
```
playwright install --with-deps
playwright test   # chromium + firefox + webkit
```
Requires `NEXT_PUBLIC_SUPABASE_URL` and `NEXT_PUBLIC_SUPABASE_ANON_KEY` repository secrets.

### Coverage Thresholds (CI fails if below)

| Package | Lines | Functions | Branches |
|---------|-------|-----------|----------|
| `@stashflow/core` | 90% | 90% | 90% |
| `@stashflow/api` | 70% | 70% | 70% |
| `apps/web` | 20% | 20% | 20% |

Thresholds enforced in `vitest.config.ts` per package — CI fails automatically.

### Remaining CI Work (P3-B)

- Automated RLS policy tests (verify cross-user reads/writes are rejected)
- Build preview deployment (Vercel preview URL on PR)
- Tighten `pnpm audit` to `--audit-level=moderate` once current advisories are resolved

## Deployment Process

### Ongoing Deployments (once configured)

| What | How |
|------|-----|
| Web | Push to `main` → CI passes → Vercel auto-deploys |
| Edge functions | `supabase functions deploy` (independent of web) |
| Schema changes | `supabase db push --linked` → `pnpm gen:types` |
| Mobile | `eas build --platform all && eas submit` (not yet configured) |

**Rollback — web:** Vercel Dashboard → Deployments → select prior → Promote to Production. Instant, no code change.

**Rollback — edge functions:** `git checkout <prior-commit> && supabase functions deploy <name> && git checkout main`

**Rollback — schema:** Write a compensating migration (never edit existing migration files).

### Phase 0 — Platform Account Setup (one-time)

Do this before any CLI work. You are collecting credentials that the later phases will use.

---

#### 0-A: Supabase

**Create account + project:**

1. Go to [supabase.com](https://supabase.com) → Sign Up (GitHub login is fastest)
2. Create an Organization (your name or team name is fine)
3. **New Project** → fill in:
   - Name: `stashflow-test` (or `stashflow-prod`)
   - Database password: generate a strong one — **save it now**, needed for CLI linking
   - Region: choose closest to you
   - Plan: Free (enough for test)
4. Wait ~2 minutes for provisioning

**Collect credentials — Settings → API:**

Navigate to your project → left sidebar → **Settings** → **API**

| What to copy | Where to find it | Save as |
|-------------|-----------------|---------|
| Project URL | "Project URL" field | `SUPABASE_PROJECT_URL` |
| Project Reference | last segment of your browser URL: `supabase.com/dashboard/project/<THIS_PART>` | `SUPABASE_PROJECT_REF` |
| `anon` public key | "Project API keys" → `anon` row | `SUPABASE_ANON_KEY` |
| `service_role` key | "Project API keys" → `service_role` row (click to reveal) | `SUPABASE_SERVICE_ROLE_KEY` |

> `service_role` bypasses RLS — treat it like a root password. Never put it in client code or commit it.

**Enable `pg_net` extension** (needed for the document webhook trigger):

Left sidebar → **Database** → **Extensions** → search `pg_net` → toggle on.

(The migration `20260506000002_documents_webhook.sql` also does `CREATE EXTENSION IF NOT EXISTS pg_net` — this just verifies it's available.)

**Create account + import repo:**

### Phase 2 — Edge Function Secrets

Supabase auto-injects `SUPABASE_URL`, `SUPABASE_ANON_KEY`, and `SUPABASE_SERVICE_ROLE_KEY` into every edge function. Only set the keys below manually.

```bash
# Generate random secrets for webhook + cron authentication
export WEBHOOK_SECRET=$(openssl rand -hex 32)
export CRON_SECRET=$(openssl rand -hex 32)

supabase secrets set \
  PARSE_LOAN_WEBHOOK_SECRET="$WEBHOOK_SECRET" \
  CRON_SECRET="$CRON_SECRET" \
  GROQ_API_KEY="<your-groq-key>" \
  GEMINI_API_KEY="<your-gemini-key>" \
  ANTHROPIC_API_KEY="<your-anthropic-key>"

# GOOGLE_VISION_API_KEY — only set if billing is enabled in Google Cloud Console
# supabase secrets set GOOGLE_VISION_API_KEY="<key>"
```

Save `WEBHOOK_SECRET` and `CRON_SECRET` — needed again in Phase 4 (pg_net trigger update).

Verify secrets are set:
```bash
supabase secrets list
```

**2.1 Deploy edge functions**

```bash

# Deploys all functions in supabase/functions/
```

### Phase 3 — pg_net Webhook Trigger

The migration `20260506000002_documents_webhook.sql` hardcodes a local Docker URL and dev JWT. After deploying to a real Supabase project, update the trigger via the Supabase SQL Editor:

**Dashboard → SQL Editor → New Query:**

```sql
-- Replace placeholders with your actual values before running
CREATE OR REPLACE FUNCTION public.tr_on_document_inserted()
RETURNS TRIGGER AS $$
BEGIN
  PERFORM net.http_post(
    url := 'https://<your-project-ref>.supabase.co/functions/v1/parse-loan-document',
    headers := jsonb_build_object(
      'Content-Type',    'application/json',
      'x-webhook-secret', '<your-PARSE_LOAN_WEBHOOK_SECRET>',
      'Authorization',   'Bearer <your-service-role-key>'
    ),
    body := jsonb_build_object('record', row_to_json(NEW))
  );
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

Substitutions:
- `<your-project-ref>` — from Settings → General
- `<your-PARSE_LOAN_WEBHOOK_SECRET>` — the `$WEBHOOK_SECRET` generated in Phase 2
- `<your-service-role-key>` — from Settings → API → service_role key

### Phase 7 — Wire CI Deploy Job

In `.github/workflows/ci.yml`, replace the commented deploy block with:

```yaml
  deploy:
    name: Deploy to Vercel
    needs: [test, security]
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 10.33.2

      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Deploy to Vercel
        working-directory: apps/web
        run: |
          npx vercel pull --yes --environment=production --token=${{ secrets.VERCEL_TOKEN }}
          npx vercel build --prod --token=${{ secrets.VERCEL_TOKEN }}
          npx vercel deploy --prebuilt --prod --token=${{ secrets.VERCEL_TOKEN }}
        env:
          VERCEL_ORG_ID: ${{ secrets.VERCEL_ORG_ID }}
          VERCEL_PROJECT_ID: ${{ secrets.VERCEL_PROJECT_ID }}
```

## Rollback Procedures

### Web

Vercel: instant rollback to any prior deployment via dashboard.

### Edge Functions

```bash
# Deploy previous function version by re-deploying from prior git ref
git checkout <prior-commit>
supabase functions deploy <function-name>
git checkout main
```

### Database

No automated rollback for applied migrations. To undo a migration:
1. Write a new migration that reverses the change
2. Apply via `supabase db push`

**Destructive migrations** (column drops, enum removals) cannot be automatically reversed — only new migrations can compensate. This is why all destructive changes should be staged: first add the new column, then deploy code that uses it, then drop the old column.

## Disaster Recovery

### Recovery Time Objectives (Target)

| Scenario | RTO Target |
|----------|-----------|
| Web app outage (Vercel) | < 30 minutes (revert deployment) |
| Edge function failure | < 1 hour (redeploy from git) |
| DB corruption | < 4 hours (restore from PITR backup) |
| Full Supabase project loss | < 24 hours (restore from backup to new project) |

### Recovery Playbook (DB Loss)

1. Create new Supabase project
2. Apply all migrations: `supabase db push --linked`
3. Restore data from latest backup
4. Update environment variables with new project URL + keys
5. Redeploy edge functions
6. Verify RLS policies are active
7. Run smoke tests

## Database Commands Reference

```bash
pnpm db:start          # start local Supabase containers (Docker required)
pnpm db:stop           # stop containers (preserves data)
pnpm db:reset          # wipe + apply all migrations from scratch
pnpm db:types          # alias for gen:types — regenerate TypeScript types
pnpm gen:types         # generate types from local schema

# New migration
pnpm supabase migration new <name>
```
